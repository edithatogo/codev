# v3.0.8 Ionic

Released: 2026-05-19

A patch on the Ionic line that fixes the most impactful regression on the multi-architect line — `afx send architect` from a builder was silently misrouted to `main` instead of the spawning sibling architect, breaking the headline feature for every adopter since v3.0.5 — and lands a substantial VSCode-extension round (per-builder file trees, View Diff, accordion mode, image paste, Backlog deduplication).

## Multi-architect routing actually works now (#774 → PR #775)

This is the embarrassing one. The sibling-architect affinity primitive shipped in v3.0.5 (#755) and the dashboard surface shipped in v3.0.6 (#761) were both behaviorally non-functional end-to-end: every builder→architect message routed to `main` regardless of which sibling architect spawned the builder. The data model, unit tests, dashboard, and in-memory architect registry were all correct — the bug lived in a single helper function that resolves the *sender's* canonical ID.

### What was broken

When `afx send architect "..."` ran from inside `.builders/<id>/`, `detectCurrentBuilderId()` used the singleton `getDb()`. From a CWD inside a builder worktree, `findWorkspaceRoot()` correctly resolves to the **worktree** (worktrees are full git checkouts with their own `codev/`), so `getDb()` opened the worktree-local `.agent-farm/state.db`. That file gets created on first DB open with the schema but has zero builder rows. The state.builders.find lookup missed, and the function fell back to the worktree directory name — `bugfix-1599` rather than the canonical `builder-bugfix-1599`.

Downstream in `tower-messages.ts`, `lookupBuilderSpawningArchitect('bugfix-1599', workspacePath)` returned `undefined` because the workspace's state.db row is keyed `builder-bugfix-1599`. With `spawningArchitect === undefined`, the routing skipped the affinity branch and fell through to the "non-builder sender" path, which routes to `main`.

The bug was invisible in single-architect workspaces (the only architect *is* main, so the wrong-but-equivalent answer looks right) and invisible in unit tests (the primitive `lookupBuilderSpawningArchitect` was tested in isolation against a populated DB — exactly the wrong scenario). It surfaced the moment an adopter (Shannon) drove a real two-architect workspace and reported "messages from ob-refine's builders are going to main."

### The fix

`detectCurrentBuilderId()` now opens the **workspace's** state.db readonly directly via a fresh `better-sqlite3` handle — mirroring the per-workspace-handle pattern already used in `lookupBuilderSpawningArchitect` (`state.ts:380`). It matches by canonical worktree path first, then a tail-segment match as a legacy-row safety net. The worktree-local state.db is no longer touched by the send path.

Five regression tests in `bugfix-774-detect-builder-id.test.ts` cover the bug scenario (worktree has its own empty state.db), the canonical-resolution happy path, both fallback branches, and the not-a-builder-worktree case. Verified end-to-end against a real two-architect workspace before tagging: `ob-refine` spawned `builder-bugfix-1599`, the builder's `afx send architect` landed on `ob-refine`'s terminal (`7e3dc4fe...`), not `main`'s (`48784891...`).

### Lesson

The v3.0.5/v3.0.6 verify phases never ran the one sentence the spec promised: "a builder spawned by ob-refine, when it sends to `architect`, reaches ob-refine." Unit tests on the primitive aren't sufficient when the feature's value prop is a specific end-to-end behavior. Going forward, release verification for any "X command does Y" claim must include literally running X and observing Y. (See [`feedback_e2e_headline_path`](/Users/mwk/.claude/projects/-Users-mwk-Development-cluesmith-codev/memory/feedback_e2e_headline_path.md) in the architect's working memory.)

## VSCode extension: per-builder diffs, accordion mode, image paste (#737 → PR #771)

Amr landed a substantial round of VSCode extension improvements. Highlights:

- **Per-builder file tree, inline.** A builder row in the Builders tree now expands to its changed-vs-default-branch files, with VSCode's native Source Control look — file-type icon plus a colored A/M/D/R/C/T status badge, tinted via your theme's `gitDecoration.*` tokens. Clicking a file opens a two-way diff (read-only merge-base base ↔ worktree file). Git is throttled to ~1 spawn / 15s per expanded builder; collapsed builders never run git. Necessary because the built-in Git decorator can't see the gitignored `.builders/` worktrees.
- **Codev: View Diff** — command + builder right-click that opens a builder worktree's whole delta in VSCode's Multi Diff Editor. Backed by a read-only `codev-diff:` provider running `git -C <worktree>` directly. Handles add/modify/delete/rename/copy and binary files.
- **Accordion mode for the Builders tree** — expanding one builder auto-collapses the others, so a reviewer never has diffs from unrelated worktrees open at once. Toggle via `codev.buildersAutoCollapse` setting (default on), a title-bar fold/unfold button, or the Command Palette.
- **Live-status dot for active builders** — a green filled dot (the running-process idiom) replaces the old `play` icon that read like a press-to-start button. Blocked builders keep the amber bell.
- **Image paste into Codev terminals** — `Cmd+Alt+V` (macOS) / `Ctrl+Alt+V` (Windows/Linux) uploads a clipboard image to Tower and injects the saved file path into the terminal. Same UX as the web dashboard and VSCode's built-in terminal; reimplemented because Codev's `Pseudoterminal`-backed terminals don't fire VSCode's built-in bridge. Per-OS clipboard read: macOS `osascript`, Linux `wl-paste`/`xclip`, Windows PowerShell. `Cmd+V` (plain text paste) is untouched.
- **Backlog hides issues that already have a builder.** An issue with an active builder no longer appears in the Backlog view nor counts toward "Backlog (N)" — matching the web dashboard. Prevents accidentally spawning a second builder for the same issue. (`hasBuilder` is machine-local; it reflects this machine's builders.)

Full per-bullet detail in `packages/vscode/CHANGELOG.md`.

## Breaking changes

None. The bugfix changes only the internal sender-ID resolution path; CLI and SDK contracts are unchanged.

## Install

```bash
npm install -g @cluesmith/codev@3.0.8
afx tower stop && afx tower start
```

If you've been running v3.0.5–v3.0.7 with multiple architects, this is the release where the feature *actually works*. The headline value prop (sibling architects receive their own builders' messages) is end-to-end verified for the first time.
